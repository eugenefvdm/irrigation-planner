Answer:

Good — you’re at the right abstraction level for MVP. Let’s structure this properly so you don’t overbuild.

⸻

1. Proper Terminology for Your Inventory

From your sketch, these look like standard plumbing/irrigation fittings:

1. Corner Sections (4x)

Correct term:
	•	90° Elbow (Slip x Slip)
	•	If threaded: 90° Threaded Elbow (MPT/FPT)

You’ll likely want:
	•	elbow_90_slip
	•	elbow_90_male_thread
	•	elbow_90_female_thread

⸻

2. Horizontal Joint (and vertical)

Correct term:
	•	Coupling (straight connector)
	•	Sometimes called Socket

If directional matters in UI:
	•	coupling_horizontal
	•	coupling_vertical

But in reality, a coupling is orientation-neutral — rotation handles direction.

⸻

3. Screw-in Joint

Correct term options:
	•	Male Adapter (MPT Adapter)
	•	Female Adapter (FPT Adapter)
	•	Threaded Coupling

If it screws into something:
	•	male_adapter
	•	female_adapter

⸻

4. 2L Flow Joiner

Likely one of these:
	•	Drip Emitter (2 L/h)
	•	Barbed Tee Connector
	•	Flow Regulated Emitter (2 LPH)

If it’s a drip irrigation output:
	•	drip_emitter_2lph

If it splits flow:
	•	barbed_tee

⸻

2. Define Your MVP Inventory Model

Start with a minimal but extensible object structure:

type PipeComponent = {
  id: string
  name: string
  category: "fitting" | "connector" | "emitter"
  ports: Port[]
  svgPath?: string
}

type Port = {
  id: string
  type: "slip" | "male_thread" | "female_thread" | "barb"
  direction: "north" | "south" | "east" | "west"
}

Example:

const elbow90: PipeComponent = {
  id: "elbow_90_slip",
  name: "90° Elbow (Slip)",
  category: "fitting",
  ports: [
    { id: "p1", type: "slip", direction: "east" },
    { id: "p2", type: "slip", direction: "south" }
  ]
}

That’s enough to support:
	•	Snap logic
	•	Valid connection validation
	•	Rotation

⸻

3. Architecture Decision: Best Platform for MVP

Since you’re web/CSS oriented, here’s the cleanest progression:

⸻

Option A (Fastest MVP): React + SVG Grid

Stack
	•	Next.js
	•	Tailwind
	•	SVG for pipe rendering
	•	Zustand for state

Why
	•	SVG gives you precision
	•	Easy drag + rotate
	•	Lightweight
	•	Deploy instantly

This is the most pragmatic choice.

⸻

Option B: React + Konva (Canvas Layer)

Library: react-konva

Pros:
	•	Built-in drag support
	•	Zoom/pan easy
	•	Better for large systems

Cons:
	•	Harder to inspect DOM
	•	Less CSS-driven

Better if you expect thousands of components.

⸻

Option C: Pure Code First (No UI)

You can bootstrap logic-only:

type GridCell = {
  x: number
  y: number
  component?: PipeComponent
  rotation?: 0 | 90 | 180 | 270
}

Build placement engine first.
Test snapping rules.
Then add UI later.

This is actually very smart if your value is layout logic.

⸻

4. Suggested MVP Phases (Strategic)

You’re building infrastructure software. Don’t start with drag and drop.

Phase 1 — Logical Engine
